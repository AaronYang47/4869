Features

Built for people who take privacy seriously.

Every feature in 4869 exists to give you absolute control over your data. From the cryptography to the user experience, nothing is compromised.

🔐

Double-Layer Encryption

Every file is encrypted with AES-256-GCM inside ChaCha20-Poly1305. Even if one cipher is ever broken, your data remains protected by the other — defense in depth at the protocol level.

🧠

Memory-Hard Key Derivation

Your password is stretched through scrypt (RFC 7914) with N=131,072, forcing 128 MB of RAM per attempt. GPU and ASIC brute-force attacks become economically prohibitive.

🎭

Plausible Deniability

Set up a decoy password that unlocks harmless placeholder files. An attacker cannot determine from the vault metadata whether the decoy or the real data was revealed.

🛡️

Post-Quantum Recovery

Optional recovery key powered by XWing-MLKEM768-X25519 hybrid KEM. If you forget your password, your data survives — and it's ready for the quantum computing era.

🗑️

Secure Deletion

Originals are overwritten with random data, truncated, then unlinked. Effectiveness depends on the storage hardware and filesystem; SSDs and copy-on-write filesystems cannot guarantee complete erasure.

📁

Native Finder Integration

Encrypted folders become .4869 packages that open directly in 4869. Double-click to unlock, just like opening any document on your Mac.

⏱️

Temporary Viewing Sessions

Peek at your files without fully unlocking. Temporary sessions remove decrypted copies after 1–30 minutes, during orderly app termination, or on the next launch after an interruption.

🔌

100% Offline

No internet connection required. No telemetry, analytics, accounts, or network requests are implemented in the app.

✅

Cryptographic Self-Test

On every launch, 4869 verifies SHA-256, AES-GCM, ChaCha20-Poly1305, and scrypt against known test vectors. If anything fails, the app refuses to operate.

Security Architecture

Cryptography you can audit.

4869 uses only well-established, peer-reviewed algorithms from Apple's CryptoKit and CommonCrypto frameworks. No custom crypto. No proprietary algorithms. No obfuscation masquerading as security.

Encryption Pipeline (Layer by Layer)

01
Password Validation
Minimum 16 characters, entropy estimation ≥ 80 bits. Rejects common, repeated, and sequential passwords.
PasswordPolicy
02
scrypt Key Derivation (RFC 7914)
N=131,072 · r=8 · p=1 · dkLen=32 bytes. Forces ~128 MB RAM per derivation attempt. With application pepper mixed into the salt.
Memory-Hard
03
Master Key Generation
64-byte cryptographically random master key generated via SecRandomCopyBytes. Never derived from or stored alongside the password.
CSPRNG
04
Key Wrapping (AES-256-GCM)
Master key sealed with the scrypt-derived key using AES-256-GCM authenticated encryption. Includes a verification check value.
AEAD
05
Per-File Key Separation (HKDF-SHA512)
Each file gets a unique sub-key derived via HKDF-SHA512 with a purpose-specific info string. Compromise of one key cannot cascade.
Key Isolation
06
Chunked Double-Layer Encryption
Files split into 4 MB chunks. Each chunk sealed with AES-256-GCM (inner), then wrapped in ChaCha20-Poly1305 (outer). Per-chunk AAD binds position and path.
Dual AEAD
07
SHA-256 Integrity Verification
Every file's plaintext hash is stored inside the encrypted manifest. Decryption fails if even a single byte is altered.
Tamper-Proof
08
Post-Quantum Recovery (Optional)
XWing-MLKEM768-X25519 hybrid KEM wraps the master key. AES-256-GCM sealed envelope with HKDF-SHA512 derived wrapping key. Resistant to Shor's algorithm.
Quantum-Safe
09
Metadata Concealment
File names, directory structure, file counts, and total sizes are encrypted inside the manifest. An attacker sees only opaque ciphertext.
Zero Leakage
10
Runtime Hardening
Core dumps disabled (setrlimit RLIMIT_CORE=0). umask 0077 for owner-only files. Mutable key buffers are zeroed on a best-effort basis.
Hardened

How It Works

Lock in seconds. Unlock in seconds.

Three steps. No accounts, no cloud, no learning curve.

1

Drop a folder into 4869

Drag any folder onto the app window, or use the file picker. 4869 scans every file, rejects symlinks, and counts the total to be encrypted.

2

Set a strong password

Choose a password of at least 16 characters. The built-in strength estimator helps you find the sweet spot between security and memorability. Optionally configure a decoy password for plausible deniability.

3

4869 encrypts, verifies, and seals

Every file is encrypted in 4 MB chunks with dual AEAD. The entire vault is then test-decrypted and verified against SHA-256 hashes before any plaintext is removed. Only after verification are originals securely shredded.

4

Double-click to unlock anytime

Your folder becomes a .4869 package with a custom locked icon. Double-click it in Finder, enter your password, and your files are restored exactly as they were — directory structure, timestamps, everything.

What a vault looks like on disk

The vault metadata (.4869.json) contains only what's necessary for decryption. File names, paths, counts, and sizes live inside the encrypted manifest — an attacker examining the metadata learns nothing about your data.

Advanced Protection

Decoy passwords and recovery keys, step by step.

These features solve different problems. A decoy password provides a separate harmless view; a recovery key provides emergency access when the real password is unavailable.

🎭

How to configure a decoy password

  1. Select an unlocked folder and enter your real password twice.
  2. Expand Plausible Deniability.
  3. Enter a different decoy password of at least 16 characters. It must not match the real password.
  4. Lock the folder normally. The current app creates a harmless README.txt placeholder for the decoy view.
  5. If you later enter the decoy password, the vault remains a .4869 package and reveals only the placeholder. Entering the real password restores the actual files.
⚠️

Decoy limitations

The decoy is not a second backup, and it does not make the existence of encrypted data disappear. It is designed so the metadata does not directly reveal whether a decoy slot is genuine, but operational clues, backups, logs, file history, or user behavior may still reveal information.

Test both passwords before relying on the feature. Never reuse either password, and never use the decoy password as the real password for another vault.

🛡️

How to generate a recovery key

  1. Use macOS 26 or later; the switch is disabled on older systems.
  2. Before locking, enable Post-Quantum Recovery Key.
  3. After encryption succeeds, choose a save location for the generated .4869-recovery-key document.
  4. Store it separately from the vault—for example, on an offline encrypted drive. Do not leave the only copy next to the vault.
  5. Keep the normal password as your primary unlock method. The recovery document is an emergency credential.
🔑

How to recover a vault

  1. Select the locked .4869 vault in 4869.
  2. Click Recovery Key instead of entering a password.
  3. Choose the matching .4869-recovery-key document.
  4. 4869 validates the key, decrypts the manifest, restores every file in staging, verifies hashes, and only then commits the restored folder.

Anyone holding both the vault and its recovery key can decrypt it. A lost recovery key cannot be regenerated from the vault. Creating a new vault generates a different key.

Each file is overwritten with cryptographically random data, truncated to zero bytes, then unlinked from the filesystem. This makes recovery with forensic tools (TestDisk, PhotoRec, Autopsy) impractical. When you set a decoy password, 4869 creates a second, hidden encrypted "slot" inside the vault metadata. If you are ever coerced into revealing your password, you can provide the decoy password instead. It will unlock a separate set of harmless files (either files you designate or a default placeholder). The vault metadata is designed so that an examiner cannot determine from its structure whether the real or decoy password was used. Even when no decoy is configured, the decoy fields are filled with random data that is indistinguishable from real encrypted content.
Defense in depth. Each file chunk is first sealed with AES-256-GCM, then the result is wrapped in ChaCha20-Poly1305. Both are authenticated encryption schemes (AEAD) — they provide confidentiality and integrity simultaneously. Using two independent algorithms means that even if a catastrophic vulnerability were discovered in one cipher, the other would still protect your data. This is the same principle behind TLS 1.3's hybrid designs.
For password-based encryption, yes. PBKDF2 is CPU-hard but not memory-hard — an attacker can build specialized hardware (ASICs) that compute millions of PBKDF2 attempts in parallel for very little cost. scrypt (RFC 7914) is both CPU-hard and memory-hard: each derivation attempt requires ~128 MB of RAM, which means an attacker would need to provision enormous amounts of memory to parallelize brute-force attacks. A 24 GB GPU can only run about 180 concurrent scrypt instances, making large-scale attacks economically impractical.
Traditional public-key cryptography (RSA, ECC, Diffie-Hellman) can be broken by a sufficiently powerful quantum computer using Shor's algorithm. 4869's optional recovery system uses XWing, a hybrid Key Encapsulation Mechanism that combines ML-KEM-768 (NIST's post-quantum standard, formerly CRYSTALS-Kyber) with X25519 (classical elliptic curve). This means the recovery key remains secure even if large-scale quantum computers become a reality. XWing is available starting from macOS 26.
4869 can encrypt any regular file — documents, photos, videos, databases, archives, source code, anything. It preserves the complete directory structure (including empty folders) and file modification timestamps. The only restriction is that symbolic links are rejected for security reasons. There is no file size limit; large files are processed in 4 MB chunks to keep memory usage flat.
4869 is designed for crash safety. During encryption, all work happens in temporary directories. The original files are not touched until every encrypted chunk has been written, verified by test-decryption, and confirmed via SHA-256. Only then are originals removed. If the app crashes at any point, the temporary data is cleaned up on the next launch, and your original files remain completely intact. The app also supports a rollback mechanism that re-decrypts committed data if plaintext cleanup fails.
